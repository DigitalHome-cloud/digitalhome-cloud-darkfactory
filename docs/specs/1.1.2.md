# DigitalHome.Cloud

## Specification 1.1.2 --- Designer Stability & Polish

**Status:** Implemented
**Previous release:** v1.1.1

------------------------------------------------------------------------

# 1. Overview

This release focuses on making the v1.1.1 Electrical Modular Ontology
**actually usable** in the SmartHome Designer. The top priority is fixing
block visibility in the toolbox — without this, no user can interact with
the electrical ontology at all. Secondary features improve label
readability, shell workflow, validation transparency, cross-app
navigation, and deployment reliability.

------------------------------------------------------------------------

# 2. Features

## 2.1 Fix: Blocks not showing in Designer toolbox (App: designer) — P0

**What:** Blocks defined by the modular ontology (v1.1.1) do not appear
in the Blockly toolbox. The designer workspace loads but the toolbox is
empty or missing electrical categories.

**Likely root causes to investigate:**

1.  **Nested toolbox subcategories** — v1.1.1 introduced nested
    categories (NFC 14-100 / NFC 15-100 inside Electrical). Blockly's
    `categoryToolbox` may not render nested subcategories depending on
    version/config. Verify this works or flatten the structure.
2.  **Silent S3 + local fallback failure** — `toolboxLoader.js` catches
    `require()` errors silently. `WorkspaceShell.js` sets
    `toolboxLoaded=true` even when `loadToolbox()` returns `null`,
    hiding the failure from the user.
3.  **Block registration error handling** — `blockRegistrar.js` has no
    try/catch. One bad block definition could break registration for all
    43 blocks.

**Details:**
-   Add proper error logging and fallback diagnostics throughout the
    block loading chain (toolboxLoader → blockRegistrar → WorkspaceShell).
-   Surface a visible error banner in the workspace when blocks fail to
    load, instead of silently showing an empty toolbox.
-   If nested subcategories are the root cause, flatten to a single
    level with prefixed names (e.g. "Electrical — NFC 15-100").

**Acceptance criteria:**
- [ ] All 43 block types visible in the Designer toolbox
- [ ] Nested or flattened categories render correctly
- [ ] Block loading errors are logged to console with actionable messages
- [ ] WorkspaceShell shows error state when toolbox load fails
- [ ] FR-DEMO shell loads with NFC 14-100 + NFC 15-100 blocks available

## 2.4 Improve block display labels for NFC circuit types (App: modeler)

**What:** Generated labels like "Nfc15100 Socket Circuit16 A 8 Sockets"
are hard to read. The PascalCase-to-label conversion doesn't handle
numbers, units, or module prefixes well.

**Details:**
-   Update `generate-blockly-toolbox.js` label generation to produce
    cleaner display names:
    -   Strip module prefix from display label (no "Nfc15100" prefix)
    -   Insert spaces before number+unit groups ("16A" → "16A", "8Sockets" → "8 Sockets")
    -   Use em-dash separator for key specs (e.g. "Socket Circuit 16A — 8 Sockets")
-   Allow label overrides via `blockly-overrides.json` for cases that
    cannot be derived algorithmically.

**Acceptance criteria:**
- [ ] Module prefix stripped from block display labels
- [ ] Number+unit groups formatted correctly
- [ ] All 20+ NFC 15-100 circuit labels are human-readable
- [ ] Override mechanism works for edge cases
- [ ] Labels remain stable for existing A-Box serialization (block type IDs unchanged)

## 2.6 Environment Namespace Routing (Apps: portal, designer, modeler)

**What:** Cross-app navigation currently uses hardcoded URLs or
`GATSBY_*_URL` env vars with production defaults. There is no runtime
namespace detection — links always point to production regardless of
whether the user is on localhost, stage, or prod. The Designer Header
even has a fully hardcoded `https://portal.digitalhome.cloud` link.

**Reference:** `docs/specs/DigitalHome-Environment-Namespace-Routing-Strategy.md`

**Core principle:** Always stay in the same namespace. Environment is a
first-class architectural dimension — navigation must respect it.

**Supported namespaces:**

| Environment | Namespace pattern | Portal | Designer | Modeler |
|-------------|-------------------|--------|----------|---------|
| DEV | `localhost:800x` | `:8000` | `:8001` | `:8002` |
| STAGE | `stage-*` | `stage-portal.digitalhome.cloud` | `stage-designer.digitalhome.cloud` | `stage-modeler.digitalhome.cloud` |
| PROD | no prefix | `portal.digitalhome.cloud` | `designer.digitalhome.cloud` | `modeler.digitalhome.cloud` |

**Implementation — shared utility `getAppUrl(appName)`:**

Create a utility function in each app (duplicated, since the apps are
separate Gatsby builds with no shared package):

1.  Detect the current namespace from `window.location.hostname`:
    -   Starts with `localhost` → DEV
    -   Starts with `stage-` → STAGE
    -   Otherwise → PROD
2.  Return the correct URL for the target app within the same namespace.
3.  Always preserve the `?home=` query parameter when navigating between
    apps so the active SmartHome context is maintained.

**Forbidden behavior — the utility must NEVER:**

-   Jump from `stage-*` to production
-   Jump from `localhost` to stage or production
-   Jump from production to stage
-   Hardcode target URLs (replace all existing hardcoded URLs)

**Files to change:**

| File | Current behavior | Change |
|------|-----------------|--------|
| `repos/portal/src/pages/index.js` | Uses `GATSBY_DESIGNER_URL` env var | Replace with `getAppUrl("designer")` |
| `repos/portal/src/components/Header.js` | Uses env var for cross-app links | Replace with `getAppUrl()` calls |
| `repos/designer/src/components/Header.js` | **Hardcoded** `https://portal.digitalhome.cloud` | Replace with `getAppUrl("portal")` |
| `repos/modeler/src/components/Header.js` | Uses `GATSBY_PORTAL_URL` env var | Replace with `getAppUrl("portal")` |
| New: `repos/*/src/utils/getAppUrl.js` | — | Create utility in each app |

**Acceptance criteria:**
- [ ] `getAppUrl(appName)` utility exists in all three apps
- [ ] Utility correctly detects DEV / STAGE / PROD namespace
- [ ] All cross-app links use `getAppUrl()` instead of env vars or hardcoded URLs
- [ ] `?home=` query parameter is preserved across cross-app navigation
- [ ] localhost:8001 → localhost:8000 (DEV stays in DEV)
- [ ] stage-designer → stage-portal (STAGE stays in STAGE)
- [ ] designer.digitalhome.cloud → portal.digitalhome.cloud (PROD stays in PROD)
- [ ] No cross-environment jumps possible

## 2.7 S3 Ontology Publishing via In-App Publisher (App: modeler) — DONE

**What:** The designer loads ontology artifacts (blockly-blocks.json,
blockly-toolbox.json) from S3 at runtime. Previously, publishing to S3
was a manual `yarn publish-ontology` step requiring local AWS
credentials, and was easy to forget.

**Problem with CI/CD approach:** Adding `publish-ontology` to the
Amplify build pipeline was attempted but the Amplify build role does not
have `s3:PutObject` permission — builds fail with "Access Denied". Rather
than managing IAM policies, we implemented an in-app solution.

**Implemented solution: In-app Ontology Publisher (`/publish/` page)**

```
Modeler Amplify build (automated):
  parse-ontology → generate-blockly-toolbox → gatsby build → deploy site
  (artifacts baked into the static site as src/data/ JSON files)

Admin publishes to S3 via /publish/ page (manual, in-app):
  S3 Browser: lists all ontology versions on S3 (visible to all users)
  Publish button: uploads JSON artifacts to S3 (admin-only, dhc-admins group)
  → ontology-graph.json, blockly-blocks.json, blockly-toolbox.json
  → uploaded to both ontology/v{VERSION}/ and ontology/latest/
  → uses Amplify Storage v6 with Cognito credentials (no IAM role needed)
```

**What is published from the browser (Option B):**
- `ontology-graph.json` — ontology graph with classes, properties, links
- `blockly-blocks.json` — block definitions for Blockly workspace
- `blockly-toolbox.json` — toolbox configuration for Blockly workspace

**What remains CI/CD or manual CLI only:**
- `context.jsonld` — not available as a static webpack import
- Module TTL files — source files authored in git, not browser artifacts
- `module-manifest.json` — published only via `scripts/publish-ontology.js`

**Files created/changed:**

| File | Change |
|------|--------|
| `repos/modeler/src/utils/s3.js` | New: Amplify Storage v6 operations (list, download, upload) |
| `repos/modeler/src/components/OntologyPublisher.js` | New: S3 browser + admin publish controls |
| `repos/modeler/src/pages/publish.js` | New: Gatsby page at /publish/ |
| `repos/modeler/src/components/Header.js` | Added "Publish" nav link |
| `repos/modeler/src/locales/*/common.json` | Added i18n keys (en, de, fr) |
| `repos/modeler/src/styles/global.css` | Added publisher CSS |
| `repos/modeler/amplify.yml` | Removed `publish-ontology` from build pipeline |

**Acceptance criteria:**
- [x] `/publish/` page accessible from modeler nav
- [x] S3 browser lists versions and files (visible to all authenticated/demo users)
- [x] Publish controls gated by `dhc-admins` group
- [x] Publish uploads JSON artifacts to both `v{VERSION}/` and `latest/`
- [x] Progress bar during upload, success/error feedback after
- [x] `publish-ontology` removed from Amplify build pipeline (no more Access Denied)
- [x] Local fallback still works for `yarn develop` without network
- [x] No regression: existing workspace save/load to S3 unaffected

# 5. Dependencies & Risks

-   **P0 bug blocks all testing** — Feature 2.1 must be resolved before
    any other feature can be verified in the designer.
-   **Blockly version compatibility** — nested category support depends
    on the Blockly version used in the designer.
-   Features 2.2 and 2.4 both modify modeler build pipeline outputs —
    coordinate to avoid merge conflicts.
-   **Feature 2.6 touches all three repos** — changes are small (one
    new utility + a few import swaps per app) but require coordinated
    commits across portal, designer, and modeler.
-   **Feature 2.7 resolved** — `publish-ontology` removed from CI/CD
    pipeline. S3 publishing is now a manual admin action via the
    `/publish/` page in the modeler app. This decouples deploy from
    publish and avoids the IAM permission issue.
-   **S3 is the single source of truth for ontology artifacts** — the
    designer must not silently fall back to stale local files in
    deployed environments. Feature 2.7 depends on 2.1 (toolbox loading
    diagnostics) for proper error surfacing.
