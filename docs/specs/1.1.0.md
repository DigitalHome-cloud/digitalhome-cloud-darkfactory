# Release v1.1.0 --- SmartHome Designer

**Status:** Draft\
**Previous release:** v1.0.0

## Goals

-   Deliver the Designer as a three-module application: SmartHome management, Blockly-based A-Box design, and 3D A-Box viewer.
-   Generate Blockly toolbox definitions from the T-Box and publish them to S3 alongside the ontology.
-   Implement domain-specific blocks for spatial, electrical, and shared design views (v1.1.0 subset of the full T-Box).
-   Enforce NF C 15-100 validation rules for French residential electrical installations.
-   Persist designs (Blockly JSON, A-Box TTL, A-Box JSON) per SmartHome ID on S3 with edit locking.
-   Extend the global library with device capability flags (actor, sensor, controller).

## Big Picture

v1.0.0 established the semantic foundation — the ontology, its visualization, and the
global library. v1.1.0 makes that foundation usable by delivering the **Designer** as the
first user-facing design tool.

The Designer application consists of three modules:

1.  **SmartHome Manager** — Create and maintain SmartHome IDs and their metadata.
2.  **Blockly Design Workspace** — Drag-and-drop A-Box design using blocks derived from the T-Box. Input: T-Box-generated block definitions. Output: A-Box TTL and JSON published to S3 under the SmartHome ID.
3.  **3D A-Box Viewer** — Visualize the SmartHome instance model (similar to the T-Box 3D graph in the modeler). The right-hand inspector shows properties of a selected item and allows attachment of catalogue items from the global library.

The Blockly toolbox is **generated from `dhc-core.schema.ttl`** at build time and published to S3 with the same versioning scheme as the ontology (`public/ontology/vX.Y.Z/`). For v1.1.0, only the spatial, electrical, and shared subsets of the T-Box are exposed as blocks; the full T-Box vocabulary will be unlocked in future releases.

### Catalogue–Equipment Relationship

A-Box equipment instances link to catalogue items (library) via a **one-to-many** relationship. A single catalogue item (e.g., an intelligent thermostat) may fulfil multiple roles — it can have actor, sensor, and/or controller capabilities. Conversely, these capabilities can be provided by separate catalogue items. Each library item must therefore declare which capability types it provides.

### Scope

-   **In scope:** SmartHome ID management, spatial blocks, electrical blocks (incl. heater), shared equipment blocks, T-Box-to-toolbox generation, NF C 15-100 validation, A-Box persistence (TTL + JSON), 3D A-Box viewer, edit locking, catalogue capability flags.
-   **Out of scope:** Network view blocks, multi-standard support (DIN VDE 0100, etc.), collaborative real-time editing, PDF export.

## Features

### SmartHome Manager (App: designer)

**What:** Module for creating and maintaining SmartHome IDs and their metadata.

**Details:**

-   Create a new SmartHome with a valid SmartHome ID (format: `{country}-{zip}-{street3letter}{housenumber}-{nn}`).
-   Edit SmartHome metadata (address, description, owner info).
-   List SmartHomes accessible to the authenticated user.
-   SmartHome ID is the partition key for all downstream data (designs, A-Box files).
-   Demo SmartHomes (`DE-DEMO`, `FR-DEMO`, `BE-DEMO`) are always visible but read-only for unauthenticated users.

**Backend changes (portal repo):**

-   `SmartHome` GraphQL type in portal backend schema (→ ADR 0011).
-   Auth rules: `dhc-users` can create and manage their own SmartHomes.

**Acceptance criteria:**

-   [ ] User can create a new SmartHome with valid ID format
-   [ ] SmartHome metadata can be edited
-   [ ] SmartHome list loads for authenticated users
-   [ ] Demo SmartHomes visible in demo mode (read-only)

### T-Box-to-Toolbox Generation (App: designer + modeler build pipeline)

**What:** Generate Blockly block definitions and toolbox configurations from the T-Box ontology, published to S3 with ontology versioning.

**Details:**

-   A build-time script (`generate-blockly-toolbox`) parses `dhc-core.schema.ttl` and produces:
    -   Block definition JSON (one entry per ontology class, with fields derived from datatype/object properties).
    -   Toolbox configuration JSON (categories derived from `dhc:designView` annotations).
-   Output is published to S3 alongside the ontology: `public/ontology/vX.Y.Z/blockly-toolbox.json` and `public/ontology/latest/blockly-toolbox.json`.
-   For v1.1.0, the generator includes a **scope filter** limiting output to the spatial, electrical, and shared design views. The filter is a configuration parameter, not hardcoded.
-   The Designer fetches the toolbox definition at runtime from S3 (versioned or latest).
-   This replaces the current hand-coded `toolbox.js` and `smartHomeToolboxes.ts` — both are deleted.

**Acceptance criteria:**

-   [ ] `generate-blockly-toolbox` script produces valid Blockly-compatible JSON from TTL
-   [ ] Generated toolbox published to S3 under `public/ontology/vX.Y.Z/`
-   [ ] Designer loads toolbox from S3 at runtime
-   [ ] Scope filter limits v1.1.0 to spatial + electrical + shared views
-   [ ] Adding a class to the T-Box automatically produces a new block (no manual coding)
-   [ ] Hand-coded `toolbox.js` and `smartHomeToolboxes.ts` removed

### Spatial Blocks (App: designer)

**What:** Blockly blocks for the spatial view — the physical structure that contains
electrical equipment.

**Blocks:**

| Block type | Ontology class | designView | Key fields |
|---|---|---|---|
| `dhc_floor` | `dhc:Floor` | spatial | `label`, `floorLevel` (integer) |
| `dhc_space` | `dhc:Space` | spatial | `label`, `areaM2` (decimal) |
| `dhc_zone` | `dhc:Zone` | spatial | `label` |

**Containment relationships:**

-   `dhc_floor` accepts a statement list of `dhc_space` children (maps to `dhc:hasSpace`).
-   `dhc_space` accepts a statement list of equipment children (maps to `dhc:hasEquipment`).
-   `dhc_space` has an optional `belongsToZone` input (maps to `dhc:belongsToZone`).

**Details:**

-   Each block exposes a `LABEL` field compatible with the existing Inspector panel.
-   `dhc_floor` uses a numeric dropdown or input for `floorLevel` (0 = ground, -1 = basement, 1 = first, etc.).
-   `dhc_space` uses a numeric input for `areaM2`.
-   Block colours follow the ontology `designView` colour scheme already defined in `dhcTheme`.

**Acceptance criteria:**

-   [ ] `dhc_floor`, `dhc_space`, `dhc_zone` blocks render in the spatial toolbox
-   [ ] Floors accept spaces as children; spaces accept equipment as children
-   [ ] `belongsToZone` connection works between space and zone
-   [ ] Inspector panel displays label, floorLevel, and areaM2 fields
-   [ ] Blocks serialise to valid Blockly JSON and can be restored

### Electrical Blocks (App: designer)

**What:** Blockly blocks for the electrical view — circuits, distribution boards,
protection devices, and electrical equipment endpoints.

**Blocks:**

| Block type | Ontology class | designView | Key fields |
|---|---|---|---|
| `dhc_distribution_board` | `dhc:DistributionBoard` | electrical | `label` |
| `dhc_circuit` | `dhc:Circuit` | electrical | `label`, `maxPoints` (integer) |
| `dhc_protection_device` | `dhc:ProtectionDevice` | electrical | `label`, `ratedCurrent` (decimal, A) |
| `dhc_socket` | `dhc:Socket` | electrical | `label`, `powerRating` (decimal, W) |
| `dhc_switch` | `dhc:Switch` | electrical | `label` |
| `dhc_light` | `dhc:Light` | electrical | `label`, `powerRating` (decimal, W) |
| `dhc_heater` | `dhc:Heater` | electrical | `label`, `powerRating` (decimal, W) |

**Containment relationships:**

-   `dhc_distribution_board` accepts a statement list of `dhc_circuit` children (maps to `dhc:hasCircuit`).
-   `dhc_circuit` has a `hasProtection` input accepting one `dhc_protection_device`.
-   `dhc_circuit` accepts a statement list of equipment endpoints: `dhc_socket`, `dhc_switch`, `dhc_light`, `dhc_heater` (maps to `dhc:feedsEquipment`).

**Details:**

-   `dhc_circuit` exposes `maxPoints` as an integer field (default: 8, per NF C 15-100).
-   `dhc_protection_device` exposes `ratedCurrent` as a dropdown with standard values (10, 16, 20, 25, 32 A).
-   Equipment blocks (`dhc_socket`, `dhc_switch`, `dhc_light`, `dhc_heater`) are subclasses of `dhc:Equipment` and can appear both inside circuits (electrical view) and inside spaces (spatial view).
-   `dhc_heater` included in v1.1.0 scope (note: `dhc:Heater` has `designView "heating"` in the T-Box but is surfaced in the electrical toolbox for this release since heating circuits are part of the electrical installation).

**Acceptance criteria:**

-   [ ] All 7 electrical blocks render in the electrical toolbox
-   [ ] Distribution board → circuit → equipment nesting works
-   [ ] Protection device attaches to circuit
-   [ ] `maxPoints` and `ratedCurrent` fields are editable and validated
-   [ ] Equipment blocks can be placed inside both circuits and spaces
-   [ ] `dhc_heater` block functions correctly alongside other equipment blocks

### NF C 15-100 Validation (App: designer)

**What:** Real-time workspace validation enforcing French residential electrical
installation rules (NF C 15-100).

**Rules to implement:**

| Rule | Description | Ontology hook |
|---|---|---|
| Max points per circuit | Lighting circuits: max 8 points. Socket circuits: max 8 points. Dedicated circuits: 1 point. | `dhc:maxPoints` on `dhc:Circuit` |
| Protection device sizing | Lighting 10A, sockets 16A, dedicated 20/32A (based on equipment type). | `dhc:ratedCurrent` on `dhc:ProtectionDevice` |
| Wire cross-section | 1.5mm² for 10A, 2.5mm² for 16/20A, 6mm² for 32A. | `dhc:crossSection` on `dhc:WiringSegment` |
| Differential protection | Every circuit group must have an RCD (30mA for residential). | [TODO] Model RCD as `dhc:ProtectionDevice` subtype or property |

**Details:**

-   Validation runs on workspace change events (debounced).
-   Violations are displayed as warnings/errors in a validation panel below the workspace.
-   Violations reference the offending block (clicking navigates to it).
-   Rules are implemented as JavaScript functions, not SHACL shapes (SHACL validation is a future modeler feature, per `semantic-core/README.md`).
-   Each rule links to the `dhc:Guideline` instance for NF C 15-100 via `dhc:governedBy`.
-   [TODO] Decide severity levels: error (blocks deployment) vs. warning (informational).
-   [TODO] Confirm the full list of NF C 15-100 rules for residential scope.

**Acceptance criteria:**

-   [ ] Exceeding `maxPoints` on a circuit shows a validation error
-   [ ] Mismatched `ratedCurrent` vs. circuit type shows a warning/error
-   [ ] Wire cross-section rule is enforced when wiring is specified
-   [ ] Validation panel lists all current violations with block references
-   [ ] Validation re-runs automatically on workspace changes

### Workspace Persistence & Edit Locking (App: designer)

**What:** Save and load SmartHome designs per SmartHome ID with explicit save and pessimistic edit locking.

**Details:**

-   Blockly workspace state is serialised to JSON (`Blockly.serialization.workspaces.save()`).
-   Additionally, the workspace is exported as A-Box TTL and A-Box JSON (derived from the Blockly state using the T-Box vocabulary).
-   All three artifacts are stored on S3 under the SmartHome ID: `public/smarthomes/{id}/design/workspace.json`, `abox.ttl`, `abox.json`.
-   Design metadata (version, last modified, locked-by) is stored via AppSync/DynamoDB in the portal backend schema (→ ADR 0011).
-   SmartHome ID is received from the portal via `?home=` query parameter.

**Save & edit locking workflow:**

-   Default mode is **view mode** (read-only).
-   User clicks **"Edit Design"** to acquire an exclusive lock. The lock is recorded in DynamoDB with the user's identity and timestamp.
-   While locked, other users see the design in view mode with a "locked by {user}" indicator and cannot enter edit mode.
-   From edit mode, the user has three actions:
    -   **Save & Release** — saves workspace + A-Box artifacts, releases the lock, returns to view mode.
    -   **Save & Keep Editing** — saves workspace + A-Box artifacts, retains the lock, stays in edit mode.
    -   **Cancel** — discards unsaved changes, releases the lock, returns to view mode.
-   Explicit save button (no auto-save in v1.1.0).
-   [TODO] Define lock timeout / stale lock recovery (e.g., lock expires after 30 min of inactivity).

**Backend changes (portal repo):**

-   `SmartHomeDesign` GraphQL type in portal backend schema with fields: `smartHomeId`, `version`, `lastModified`, `lockedBy`, `lockedAt`.
-   Mutations: `lockDesign`, `unlockDesign`, `saveDesign`.
-   Auth rules: `dhc-users` group can read/write designs for SmartHomes they have access to.

**Acceptance criteria:**

-   [ ] Workspace opens in view mode by default
-   [ ] "Edit Design" acquires lock and enables editing
-   [ ] Locked design shows "locked by {user}" to other users
-   [ ] "Save & Release" saves all artifacts and releases lock
-   [ ] "Save & Keep Editing" saves without releasing lock
-   [ ] "Cancel" discards changes and releases lock
-   [ ] A-Box TTL and JSON are written to S3 under the SmartHome ID
-   [ ] Unauthenticated users (demo mode) see view mode only, cannot lock
-   [ ] SmartHome ID from `?home=` parameter is used as the partition key

### 3D A-Box Viewer (App: designer)

**What:** 3D force-directed graph visualization of the SmartHome instance model, similar to the T-Box viewer in the modeler.

**Details:**

-   Renders the A-Box graph (loaded from `abox.json` on S3) as a 3D force-directed graph.
-   Reuses the modeler's 3D graph component (`three.js` / `3d-force-graph`) adapted for A-Box instance data.
-   Nodes represent A-Box instances (spaces, circuits, equipment, etc.), coloured by `designView`.
-   Edges represent object property relationships (`hasSpace`, `hasCircuit`, `feedsEquipment`, etc.).
-   **Right-hand inspector panel** shows:
    -   Properties of the selected instance (label, datatype property values).
    -   Attached catalogue item(s) from the global library.
    -   Action to attach/detach catalogue items to the selected equipment instance.
-   Catalogue attachment is persisted as part of the A-Box (via `dhc:hasEquipmentType` / `dhc:catalogRef`).

**Acceptance criteria:**

-   [ ] A-Box graph renders as a 3D force-directed graph
-   [ ] Nodes are coloured by design view
-   [ ] Selecting a node shows its properties in the inspector
-   [ ] Catalogue items can be attached to equipment instances via the inspector
-   [ ] Catalogue attachment is persisted in the A-Box

### Library: Device Capability Flags (App: modeler)

**What:** Extend library items with capability flags indicating whether a catalogue item provides actor, sensor, and/or controller functions.

**Details:**

-   Each library item gains three boolean fields: `hasActorCapability`, `hasSensorCapability`, `hasControllerCapability`.
-   This reflects the real-world fact that a single device (e.g., an intelligent thermostat) may combine sensor (temperature reading), actor (valve control), and controller (logic/scheduling) in one unit — or these may be separate devices.
-   The capability flags are used by the 3D A-Box viewer to show what roles a catalogue item fulfils when attached to equipment.
-   Requires a schema update to the `LibraryItem` GraphQL type in the portal backend.

**Acceptance criteria:**

-   [ ] Library item form includes actor/sensor/controller capability checkboxes
-   [ ] Capability flags are stored and displayed in the library list
-   [ ] 3D viewer inspector shows capability flags for attached catalogue items

## Open Questions

-   [ ] **Equipment type UX:** How should users select equipment types from the global library — dropdown, block connection, or search panel?
-   [ ] **RCD modelling:** Model RCD as a subclass of `dhc:ProtectionDevice`, a separate class, or a property on `dhc:ProtectionDevice`?
-   [ ] **Validation severity:** Should NF C 15-100 violations be errors (block save/export) or warnings (informational)?
-   [ ] **Full NF C 15-100 scope:** Which additional residential rules beyond max-points, protection sizing, and cross-section should be included?
-   [ ] **Wiring blocks:** Should `dhc_wiring_segment` be a visible block in v1.1.0, or is cross-section inferred from the protection device rating?
-   [ ] **Lock timeout:** How long before a stale edit lock is automatically released (e.g., 30 min inactivity)?
-   [ ] **SmartHome access model:** How is SmartHome ownership / access control modelled? Can a user access multiple SmartHomes? Can a SmartHome have multiple users?
-   [ ] **Toolbox generation — property mapping:** How should T-Box object properties map to Blockly connection types (statement input, value input, dropdown)? Define the mapping rules for the generator.

## Dependencies & Risks

-   **Portal backend changes required.** New GraphQL types (`SmartHome`, `SmartHomeDesign`, `LibraryItem` update) need schema changes + `amplify push`. Coordinate with portal repo. Document backend ownership decision in ADR 0011.
-   **Ontology stability.** All blocks map to `dhc-core.schema.ttl` v1.0.0 classes. No ontology changes are planned, but if NF C 15-100 modelling requires new classes (e.g., RCD subtype), a v1.0.1 ontology patch may be needed.
-   **Broken SmartHomeDesigner path.** The existing `SmartHomeDesigner.tsx` has a broken import (`../ontology-designer/WorkspaceShell`). Must be fixed as part of toolbox consolidation.
-   **Toolbox generator complexity.** Generating Blockly blocks from TTL is a new build pipeline step. Mapping T-Box constructs (class hierarchy, property domains/ranges, cardinality) to Blockly block shapes requires clear rules.
-   **3D viewer reuse.** Adapting the modeler's T-Box 3D graph for A-Box instances may require refactoring the component to be data-source-agnostic.
-   **Blockly version.** Verify that the installed Blockly version supports all serialisation features needed for persistence.
-   **Demo SmartHomes.** `FR-DEMO` should have a pre-built example design showcasing NF C 15-100 compliance.
